 DOSSIER D'ARCHITECTURE : Environnement de Sécurité Ce document a pour but de vous aider à situer les concepts de sécurité appris dans un environnement technique concret. Il fait le pont entre la théorie (ce qu'on veut protéger) et la pratique (comment on le protège). 1. Architecture Logique des Fonctions de Sécurité Le laboratoire met en œuvre plusieurs couches de défense en profondeur. Chacune correspond à une fonction logique implémentée pour contrer des vulnérabilités spécifiques : Confidentialité (Passerelle TLS) : Interception et chiffrement des flux pour éviter l'écoute clandestine ( sniffing ). Contrôleur d'Accès (PDP/PEP) : Découplage de l'autorisation de l'application via un contrôle d'accès basé sur les rôles (RBAC). Validation Applicative ( AppSec ) : Assainissement strict des entrées utilisateur pour neutraliser l'injection de commandes. Ségrégation Réseau : Séparation étanche des périmètres pour empêcher les mouvements latéraux post-compromission. Diagramme Logique : Extrait de code 2. Virtualisation des Fonctions (NFV) et Docker La virtualisation des fonctions réseau (NFV) consiste à remplacer des boîtiers de sécurité matériels (Firewalls physiques, Appliances VPN) par des conteneurs logiciels. Dans ce Lab , Docker est utilisé pour implémenter ce paradigme. Chaque "boîtier" est un conteneur immuable. Méta-Modèle du concept : Extrait de code 3. Architecture Physique (Composants et Interfaces) Voici comment les conteneurs (composants physiques) sont répartis sur les réseaux virtuels, ainsi que les interfaces qu'ils exposent : demo_wireshark : Le client/attaquant. Interface : Sorties cURL . demo_serveur (Nginx) : Passerelle TLS. Interface : Ports 80 (HTTP) et 443 (HTTPS). demo_opa : Moteur d'autorisation. Interface : Port 8181 (API REST POST). app_vulnerable : Serveur Web Python. Interface : Port 8080 (HTTP). app_database : Serveur Backend isolé. Interface : Port 5432 (TCP interne).  INSTRUCTIONS PAR LABORATOIRE Dans la barre verte en haut de Wireshark (la barre "Apply a display filter" ), tapez les filtres suivants selon le laboratoire en cours, puis appuyez sur Entrée . 🔓 LAB 1 : Sécurité des Échanges (TLS) - L'exercice principal pour Wireshark L'objectif : Prouver que sans chiffrement, tout est lisible. Nettoyez Wireshark (Bouton 🔄 -&gt; Continue without saving ). Le Filtre à appliquer : Tapez simplement http et faites Entrée. L'Action : Dans le Terminal Web, lancez la commande en clair : curl -X POST -d "password=MotDePasseSecret123" http://demo_wireshark L'Analyse : Regardez Wireshark. Cliquez sur la ligne "POST" qui vient d'apparaître. Dans le panneau du bas (Packet Details), déroulez la ligne "HTML Form URL Encoded" . Le mot de passe est visible ! Maintenant la version sécurisée : Nettoyez Wireshark à nouveau ( 🔄 -&gt; Continue without saving ). Si le signe rond n’est pas visible, cliquer sur le bouton ci-dessus encercle en rouge Le Filtre à appliquer : Tapez tls (pour Transport Layer Security). L'Action : Lancez la commande chiffrée : curl -k -X POST -d "password=MotDePasseSecret123" https://demo_wireshark L'Analyse : Regardez Wireshark. Vous ne voyez plus de "HTTP". Vous voyez "Application Data". Le contenu est mathématiquement illisible. LAB 2 : Sécurité Réseau (Micro-segmentation) L'objectif : Voir les paquets de ping ("Echo Request"). La Micro-segmentation Réseau (Isolation des zones) Le Concept : Dans une entreprise, on ne met pas tous les serveurs dans le même panier. On sépare le "Front" (accessible depuis Internet) du "Back" (les données sensibles). Ici, le conteneur Wireshark (le pirate) est sur le réseau front-net. La base de données (app_database) est cachée dans le réseau back-net. 1. Préparation dans Wireshark Nettoyez la capture (Bouton 🔄 -&gt; Continue without saving ). Appliquez le filtre : icmp (C'est le protocole réseau utilisé par la commande ping). 2. ❌ Le Cas "Mouvement Lat é ral" (Ce qui se passe sans isolation) Si un pirate réussit à entrer sur le réseau public (front-net), il va chercher à voir quelles autres machines s'y trouvent pour rebondir dessus. Dans le terminal Ubuntu , tapez le ping vers le serveur web vulnérable (qui est sur le même réseau) : Bash docker exec demo_wireshark ping -c 2 app_vulnerable Le Résultat dans le terminal : Le ping réussit ! Les paquets s'échangent normalement (64 bytes from...). Cela montre aux étudiants qu'une fois dans le réseau Front, un pirate peut se déplacer librement d'une machine à une autre. Dans Wireshark : Vous allez voir apparaître des lignes bleues/rose avec écrit Echo (ping) request et Echo (ping) reply. Le trafic réseau est totalement visible. 3. ✅ Le Cas "Micro-segmentation R é ussie" (La S é curit é ) Maintenant, le pirate essaie d'atteindre sa vraie cible : la base de données de l'entreprise (app_database), qui contient les données bancaires ou les utilisateurs. Dans le terminal Ubuntu , tapez le ping vers la base de données : docker exec demo_wireshark ping -c 2 app_database Le Résultat dans le terminal : Le terminal affiche immédiatement : ping: bad address 'app_database' (ou reste bloqué sans réponse). L'accès est totalement impossible. Dans Wireshark : Rien du tout ! Écran blanc. Aucun paquet icmp n'apparaît. La conclusion pédagogique à montrer aux étudiants : C'est la preuve par l'action de l'efficacité de la sécurité réseau. Même si le conteneur demo_wireshark (le pirate) utilise les meilleurs outils du monde, le cloisonnement des réseaux Docker l'empêche techniquement d'aiguiller ses paquets vers la base de données. Le cœur du système est à l'abri ! LAB 3 : Gestion des Accès avec OPA (Vérification) 1. Préparation dans Wireshark Nettoie la capture (Bouton 🔄 ). Applique le filtre exact : tcp.port == 8181 (assure-toi que la barre devienne bien verte et qu'il n'y ait pas de lettres en trop à la fin) . 2. ❌ Le Cas "Pas OK" (Acc è s Refus é ) On envoie un utilisateur qui a le rôle guest. Ton script ne le connaît pas, il doit le bloquer. Commande à taper dans ton terminal Ubuntu : Bash docker exec demo_wireshark curl -X POST http://opa:8181/v1/data/system/authz/allow \ -H "Content-Type: application/json" \ -d '{"input": {"user": {"role": "guest"}}}' Le Résultat dans le terminal : OPA te répond directement : JSON {"result":false} Dans Wireshark : Tu verras une ligne HTTP POST. Clique dessus, déroule la section JavaScript Object Notation tout en bas de la capture. Tes étudiants verront le JSON contenant {"role": "guest"} et la réponse d'OPA qui dit non. 3. ✅ Le Cas "OK" (Acc è s Autoris é - Admin) On envoie un administrateur. Ton premier bloc allow if doit s'activer. Commande à taper dans ton terminal Ubuntu : Bash docker exec demo_wireshark curl -X POST http://opa:8181/v1/data/system/authz/allow \ -H "Content-Type: application/json" \ -d '{"input": {"user": {"role": "admin"}}}' Le Résultat dans le terminal : OPA valide et répond : JSON {"result":true} 4. 🔥 Le Cas "OK" (Accès Autorisé - Employé sur données publiques) C'est la force de ton script ! Un employé a le droit d'entrer UNIQUEMENT s'il fait un GET sur /donnees-publiques. Commande à taper dans ton terminal Ubuntu : Bash docker exec demo_wireshark curl -X POST http://opa:8181/v1/data/system/authz/allow \ -H "Content-Type: application/json" \ -d '{"input": {"user": {"role": "employee"}, "request": {"method": "GET", "path": "/donnees-publiques"}}}' Le Résultat dans le terminal : OPA valide le deuxième bloc de ton script et répond : JSON {"result":true} Exercice bonus: GET par POST et /donnees-publiques par /les-salaires dans la commande de l'employé. Donnez les 2 résultats (captures d’écrans) et justifier les. C’est une façon de vous montrer la puissance d'un moteur de politique dynamique ! LAB 4 : Sécurité Applicative (AppSec / Injection de Commande) Le Concept : L'application a été mal codée. Elle prend une information de l'utilisateur (une adresse IP) et l'exécute directement sur le serveur sans la nettoyer. Préparation dans Wireshark Nettoyez la capture ( 🔄 ). Appliquez le filtre : tcp.port == 8080 (Le port de votre application Python vulnérable). ✅ Le Cas "OK" (Utilisation L é gitime) U n utilisateur normal utilise l'application pour ce qu'elle est censée faire : un simple ping. Dans le terminal Ubuntu , tapez : Bash docker exec demo_wireshark curl "http://app_vulnerable:8080/ping?ip=127.0.0.1" Le Résultat dans le terminal : L'application répond normalement avec les statistiques du ping. Tout va bien. Dans Wireshark : Vous voyez une simple requête GET /ping?ip=127.0.0.1 HTTP/1.1. C'est un trafic web totalement propre. ❌ Le Cas "Pas OK" (La Vuln é rabilit é Exploit é e) Le pirate (vous !) injecte des commandes Linux à la suite de l'adresse IP en utilisant le point-virgule ;. Dans le terminal Ubuntu , tapez l'attaque (celle qui a marché tout à l'heure !) : docker exec demo_wireshark curl "http://app_vulnerable:8080/ping?ip=127.0.0.1;whoami;cat+/etc/passwd" Le Résultat dans le terminal : Catastrophe. Vous voyez root et tout le fichier des mots de passe /etc/passwd. Le serveur est compromis. Dans Wireshark : Cliquez sur le paquet HTTP rouge/vert. Déroulez l'en-tête Hypertext Transfer Protocol . L'étudiant peut voir exactement à quoi ressemble une attaque sur le réseau : l'URL contient la charge virale (payload) ;whoami;cat+/etc/passwd. C'est comme ça que les pare-feux applicatifs (WAF) repèrent les attaques ! Étape 1 : Le Correctif de Sécurité (Le Code Propre) Pour corriger l'application, nous allons modifier son code Python (généralement un fichier nommé app.py ou similaire dans votre dossier) pour y intégrer la bibliothèque ipaddress de Python. Elle va vérifier que ce que saisit l'étudiant est uniquement et strictement une adresse IP valide avant de lancer le ping. Ouvrez le fichier de votre application Python (par exemple nano app.py) et modifiez la fonction qui gère le ping pour qu'elle ressemble à ceci : Python import subprocess import ipaddress from flask import Flask, request, jsonify app = Flask(__name__) @app.route('/ping', methods=['GET']) def ping(): user_ip = request.args.get('ip', '') # --- SCÉNARIO DE PROTECTION : VALIDATION STRICTE --- try: # Si ce n'est pas une vraie IP, cette ligne lève une erreur immédiatement ipaddress.ip_address(user_ip) except ValueError: # Si le pirate a mis un ";" ou du texte, on bloque tout de suite ! return jsonify({"error": "Attaque détectée ou IP invalide ! Saisie refusée."}), 400 # ---------------------------------------------------- # Si c'est une vraie IP, on exécute le ping de manière sécurisée res = subprocess.run(["ping", "-c", "1", user_ip], capture_output=True, text=True) return res.stdout Une fois le fichier sauvegardé, redémarrez le conteneur de l'application pour appliquer le patch : Bash docker compose up -d --force-recreate app_vulnerable 🧪 Étape 2 : Vérification du Scénario "Sécurisé" (OK) On demande d'abord à l'étudiant de tester le comportement normal du système après la mise à jour de sécurité. Dans le terminal Ubuntu : Bash docker exec demo_wireshark curl "http://app_vulnerable:8080/ping?ip=127.0.0.1" Le constat : L'application valide l'IP 127.0.0.1 et répond avec les statistiques du ping. Tout fonctionne pour l'utilisateur légitime. 🛡️ Étape 3 : Vérification face à l'Attaque (Le Blocage) Maintenant, le pirate tente d'exécuter à nouveau son attaque d'injection de commande. Dans le terminal Ubuntu : Bash docker exec demo_wireshark curl "http://app_vulnerable:8080/ping?ip=127.0.0.1;whoami;cat+/etc/passwd" Le constat dans le terminal (Sécurisé) : L'application ne se fait plus avoir. Elle renvoie instantanément : JSON {"error": "Attaque détectée ou IP invalide ! Saisie refusée."} Le fichier /etc/passwd n'est plus du tout affiché, et la commande whoami n'a jamais été exécutée. Le serveur est sain et sauf ! Le constat dans Wireshark : En filtrant sur tcp.port == 8080, cliquez sur la réponse HTTP renvoyée par le serveur. Vous verrez le code de retour HTTP 400 Bad Request au lieu du code 200 OK. 💡 Conclusion pédagogique pour vos étudiants Avec cette manipulation, vos étudiants comprennent le principe fondamental de l' AppSec (Sécurité Applicative) : « Never trust user input » (Ne faites jamais confiance aux entrées utilisateur). La sécurité ne dépend pas toujours du réseau, elle se joue aussi directement dans les lignes de code écrites par les développeurs !
